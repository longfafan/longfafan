在本单元中，你学习了 UIKit 框架提供的用于管理 app 流程的机制。先前你学习了如何使用自动布局和堆栈视图管理视图以及控件的位置和大小。现在，将所有知识结合起来构建一个 app。

在此指导项目中，你要创建一个性格测试。可能你以前看到过此类游戏。玩家看到的是轻松的主题，他们回答的问题会对应到特定的结论。这里是一些性格测试主题的示例：

  • 你是什么动物？
  • 接下来你应该去哪个国家旅行？
  • 你是哪种 Apple 产品？

测试问题没有正确答案。答案都是主观的，结果甚至可以不符合逻辑。例如，假定你设计的测试称为“接下来你应该去哪个国家旅行？”。你可以使用问题“你最喜欢什么颜色？”，并决定答案“绿色”对应意大利而不是法国。其他问题和答案可能更有实际意义。如果玩家喜欢寿司胜过意面，那么你可以增加日本的分数，而不是意大利。

此指导项目将使用“你是哪种动物？”作为测试主题。四种可能的结果分别是：狗、猫、兔子、乌龟。但如果你更愿意自己选择主题和结果，也没问题。只要你按照项目中的步骤执行，任何主题都可以。对于这个过程，如果你乐在其中，你就能学到更多。

 

第 1 部分

项目规划

此时无需直接深入了解 Interface Builder 或 Swift，重要的是先考虑一下性格测试的目标和要求。目标受众是谁？可能你心中已经有了主题，但测试要包括哪些功能呢？要实现这些功能，需要有哪些模型和视图呢？

如果你还没想清楚这些问题就尝试深入学习编写代码，那最后可能得重写或者浪费很多工作。现在花些时间规划项目，考虑最佳实现方式。

功能特色

要做出一个有趣的性格测试，需要哪些功能呢？因为这可能是你编写的第一个此类 app，所以不要有太多功能。此 app 至少应完成三个主要目标：

  1. 向玩家介绍此测试。
  2. 显示问题和答案。
  3. 显示结果。

工作流程

记住这三个目标，尝试将 app 想象成一系列相关视图。每个功能都跟另外两个不同，所以每个功能都要有自己的屏幕。有些输入类型会让玩家从一个功能移动到下一个功能。例如，可以包含一个“开始测试”按钮，用于将玩家从简介移动到问题，而回答完最后一个问题之后，就可以转换为显示结果。为了实现此项目的目的，至少需要三个视图控制器，分别用于显示这三个功能。

你是不是也考虑过要为每个问题加载一个新的视图控制器？其实没必要。在之前的课程中，你学过两种显示视图控制器的方法：模态或者压入导航堆栈。模态显示的屏幕中通常包括一种关闭方式，而压入导航控制器的所有新视图控制器的左上角都有一个“返回”按钮。无论哪种情况，总有一种方法可以执行关闭。

如果每个问题都有不同的视图控制器，那会怎样？假设玩家处在第九个问题，想要返回第三个问题。如果每个问题都有一个视图控制器，那么玩家得关闭六个视图控制器才能返回。这也还可以，可是随后，他们还要重新回答一遍中间的所有问题才能返回到第九个问题。

在此项目中，你要更新一个可以显示所有问题的屏幕，而不是每个问题都用单独的屏幕显示。

 

视图

性格测试可能需要不同的输入控件，具体取决于测试主题和你要提出的问题。思考一下以下问题：

  • 你最喜欢哪种食物？
  • 以下哪种食物是你喜欢的？
  • 你有多喜欢这种特定食物？

第一个问题是选择题，只有一个答案有效。对于此问题，可以每种食物使用一个按钮。第二个问题可以有零个答案，也可以有多个答案。你可以使用多个开关让玩家能随意选择多种食物，还要有一个用于提交选择结果的按钮。第三个问题可能涉及到使用滑块的 1 到 10 的标尺。

如你所见，问题的类型决定了要在屏幕上显示的控件。如果测试中为所有问题和答案使用一个视图控制器，那么你可能希望只显示与当前问题匹配的按钮、开关、滑块和/或标签。要执行此操作，最简单也是最好的方式是将控件分组到问题类型所对应的视图中，然后可以显示或隐藏单个答案、多个答案或范围响应的相应视图。

模型

你的性格测试需要哪种类型的数据？首先，你可能会考虑创建字符串的数组用来容纳问题，但答案储存在哪里呢？有一种更好的方法，就是创建用于容纳答案集合的 Question 结构体。答案集合也不能只是一个字符串数组，因为每个答案都对应一种测试结果。最少也要包含一个 Question 结构体、一个 Answer 结构体和某种结果类型。

思考一下结果类型。在性格测试中，一个答案只能对应一种结果。例如，在动物测试中，结果永远不会是狗和猫，而只能是其中之一。这是枚举的一个经典用例。就像 Direction 枚举可能有 north、south、east、west 情况，同样，AnimalType 可以是 dog、cat、rabbit 或 turtle。要查看使用 enum 的详细信息，请访问本单元中先前部分的“枚举”一课。

 

快速概览

既然已经分析过需要哪些组件了，那么接下来看看如何将项目组合起来，这可能要简单一些。你会为该测试使用三个视图控制器：

  • 第一个是简介屏幕，其中包含有关测试的信息和开始按钮。
  • 第二个视图控制器显示一个问题和几个答案，并可以管理响应。此视图控制器会针对每个问题刷新一次，并且会根据问题种类来显示相应的控件。
  • 第三个视图控制器会统计所有答案并显示最终结果。该结果可以关闭，以便下一个玩家从第一个视图控制器开始进行测试。

创建问答屏幕

第二个视图控制器将显示每个问题（一次显示一个），以及输入控件（让玩家可以做出响应）。你使用的控件（按钮、开关或滑块）对测试中的问题和答案而言应该是有意义的。在项目的稍后部分中，你要深入思考一下这些控件。

玩家回答完问题以后，app 需要做出决定：

  • 如果测试中还有别的问题，就要相应地更新视图控制器中的标签和控件。
  • 如果没有别的问题了，就要在新的视图控制器中显示结果。

A pp 怎么知道要做什么？你需要创建一些逻辑，用于确定在收到当前问题的答案后是否执行过渡。如果你创建过渡的方式是按住 control 键并从输入控件拖移到下一个视图控制器中，那么只要玩家与该控件交互，就会强制执行过渡。

你可以改为采用编程方式在第二个和第三个视图控制器之间调用过渡。按住 control 键并从视图控制器图标（位于第二个视图控制器的视图顶部）拖移到第三个视图控制器中，然后创建 Show 过渡。在 Storyboard 中高亮显示该过渡。然后使用“Attributes”检查器为其提供标识符字符串“Results”。

 

你已经学过，在将根视图控制器嵌入导航控制器时，Show 过渡将从模态显示调整为从右到左压入。在你的测试中，当玩家回答完最后一个问题时，你可以压入最后一个视图控制器以显示结果。

三个视图控制器是否应该都包含在导航控制器中 ？请记住，无论何时，只要 app 的上下文会更改，使用模态显示就是正确的选择。当玩家从简介屏幕转换到问题屏幕时，就是上下文转换。这意味着，第一个视图控制器应该模态显示包含在导航控制器中的视图控制器。选择第二个视图控制器，然后点按“Embed In”按钮，并选择导航控制器。

创建从第一个视图控制器的按钮到导航控制器的 Show 过渡。这样，用户就能开始测试。

创建描述性子类

重复这些步骤以创建第二个类，将其命名为“ResultsViewController”。完成之后，你会在项目导航器中看到两个关于你的测试的新文件 。

你可能还会注意到，项目导航器中列出了名为“ViewController”的 UIViewController 的子类。“Single View App”模板自动为 app 的第一个视图控制器指定了此名称。为了具有更强的描述性，请点按文件名并将其更改为“IntroductionViewController.swift”。接下来，打开文件并将类名更改为“IntroductionViewController”，然后关闭文件。

现在项目有三个描述性命名的 UIViewController 子类。重新打开 Main.storyboard。一次打开一个，选择每个视图控制器，并使用“Identity”检查器为其分配适当的自定类。第一个视图控制器将为 IntroductionViewController，接着是 QuestionViewController 和 ResultsViewController。

 

第 3 部分

创建问题和答案

在项目规划阶段，你已经考虑过三种不同的问题类型：单个答案、多个答案和范围响应。现在花些时间做出问题列表。考虑一下每个问题如何措辞，以将其归入三种类别之一。

单个答案的问题

假设你提问“你最喜欢哪种食物？”答案可能列出四种食物，玩家必须选择一项。你会使用哪种控件呢？简单的方法是以垂直堆栈视图的组织方式显示每个答案的按钮。

开始时，将垂直堆栈视图从“Object”库中拖移到 QuestionViewController。现在为堆栈添加 4 个按钮。使用“Align”工具将堆栈在视图中垂直居中，然后使用“Add New Constraints”工具将其左边距和右边距设置为 20 像素。在“Attributes”检查器中将“Spacing”设置为 20，在按钮之间添加空隙。如果需要，请点按“Update Frames”按钮，以根据创建的约束重新安排堆栈。

 

当然，按钮标题会根据你提供的答案而改变。稍后在你对测试编写代码时，再更新按钮标题。

多个答案的问题

问题“以下哪些食物是你喜欢的？”表明玩家可以选择多个答案。此时无需使用答案按钮，更适合的方法是创建标签和开关对，因为这样玩家就可以打开所有肯定的答案。玩家在做出选择之后，可以轻点按钮以提交答案并继续下一个问题。

 

 

在开始之前，你可能想要隐藏单个答案堆栈视图。在 Storyboard 或大纲中选择堆栈，然后打开“Attributes”检查器，取消选择面板底部的“Installed”。

开关 UI 跟按钮 UI 相差无几。水平堆栈视图中可以容纳每个标签和开关对。跟单个答案问题一样，也可以在垂直堆栈视图中容纳行。

开始时，从“Object”库中添加标签和开关。高亮显示这两者，然后点按“Embed In”按钮并选择“Stack View”。在堆栈视图的“Attributes”检查器中，务必确认已将“Axis”设置为“Horizontal”，“Alignment”和“Distribution”均设置为“Fill”。

 

选择该堆栈视图，然后拷贝 (command-C) 并粘贴 (command-V) 以将三个副本添加到视图中。现在选择全部四个水平堆栈，然后点按“Embed In”按钮并选择“Stack View”以将其放在另一个堆栈视图中。在此新堆栈视图的“Attributes”检查器中，将“Axis”设置为“Vertical”，“Alignment”和“Distribution”都设置为“Fill”，元素之间的“Spacing”设置为 20。

 

 

在堆栈视图的底部添加一个按钮，并将其标题设置为“Submit Answer”。最后，使用“Align”工具将堆栈放在视图中垂直居中的位置，然后使用“Add New Constraints”工具将左边缘和右边缘设置为距每个页边空白 20 像素。如果需要，请点按“Update Frames”按钮，以根据创建的约束重新安排边框。

 

 

范围问题

第三种问题类型格式可能类似于：“你有多喜欢这种食物？”你也许能想到让答案使用按钮或开关的方法，但如果玩家在选择时自由度更高，可能体验会更好。为了让玩家能选择某个范围内的答案，你可以使用滑块作为输入控件，滑块两端各有一个标签。

 

为了更简单起见，你可能希望隐藏多个答案堆栈，就像先前隐藏单个答案堆栈时一样。在 Storyboard 中将其选中，打开“Attributes”检查器，然后取消选中面板底部的“Installed”。

 

你可以使用堆栈视图来创建此界面，而不必定义很多约束，这就跟使用开关的方法类似。开始时，将“Object”库中的两个标签添加到画布上，然后将两个标签选中，点按“Embed In”按钮，并选择“Stack View”。在“Attributes”检查器中，检查是否已将“Axis”设置为“Horizontal”，“Alignment”设置为“Fill”，“Distribution”设置为“Equal Spacing”。

 

 

接下来，从“Object”库中拖移一个滑块到画布上。选择滑块和水平堆栈，然后点按“Embed In”按钮并选择“Stack View”。在“Attributes”检查器中，检查是否已将“Axis”设置为“Vertical”，“Alignment”和“Distribution”均设置为“Fill”，“Spacing”设置为 20。

 

在堆栈视图的底部添加一个按钮，并将其标题设置为“Submit Answer”。使用“Align”工具将堆栈放置在视图中垂直居中的位置，然后使用“Add New Constraints”工具将左边缘和右边缘设置为与每个页边空白的间距为 20 像素。如果需要，请使用“Update Frames”按钮，以根据创建的约束重新安排边框。

 

你需要先重新启用在构建过程中卸载的堆栈视图，然后再继续。在文稿大纲中，选择每个堆栈视图，然后在“Attributes”检查器中选中“Installed”复选框。

问题标签和进度

无论你向玩家提出哪种问题，都需要将问题显示在标签中。在视图的顶部添加一个标签。使用“Add New Constraints”工具将标签放置在导航栏下方 20 像素、距离左页边空白和右页边空白 20 像素的位置。在“Attributes”检查器中，将文本设置为居中对齐，将字体设置为“System 32.0”。将“Lines”属性设置为 0，以允许标签按需使用任意数量的行。将“Line Break”属性更改为“Word Wrap”。

 

 

玩家一般都想知道自己已经完成了测试中的多少问题。在“Object”库中搜索“Progress View”，然后将其添加到视图。使用“Add New Constraints”工具将进度视图定位在距底部 20 像素、距视图左边缘和右边缘 20 像素的位置。

 

 

第 4 部分

模型和插座变量

本课程到目前为止，你已经在 Storyboard 中设计了视图控制器，并且已经准备好了 3 个 UIViewController 子类来接收某些代码。现在是时候开始创建容纳问题数据的结构体了，并且根据每个问题的值及其答案来更新用户界面。在完成数据布局之后，即可根据显示的问题来更新用户界面。

数据模型

创建名为 Question.swift 的新文件以存储模型定义。使用此文件来定义性格测试所需的所有结构体。你可以通过在 Xcode 菜单栏中选择“File”>“New”>“File”（或 command-N），然后选择“Swift file”来创建此文件。

完全可以假定每个 Question 都有用于表示问题本身的文本，还有 Answer 对象的数组。因为你的测试可以使用三种不同类型的输入方法，所以你要创建一个 enum，用来描述问题的响应类型：单个答案、多个答案或范围响应。如下所示是结构体的示例：

 

struct Question {

var text: String

var type: ResponseType

var answers: [Answer]

}

 

enum ResponseType {

case single, multiple, ranged

}

 

每个答案都对应一种结果类型。在动物的示例中，假设你提问“以下哪种食物是你最喜欢的？”，答案可能是：“牛排”、“鱼”、“胡萝卜”、“玉米”。每个响应分别对应狗、猫、兔子、乌龟，所以也就对应特定的表情符号。如果 answers 属性是字符串的数组，那么没有什么简单的方式可以将答案与特定结果相关联。Answer 结构体会有一个向玩家显示的字符串和一个 type 属性，该属性将答案与特定结果绑定在一起。

这里是数据的示例：

 

struct Answer {

var text: String

var type: AnimalType

}

 

enum AnimalType: Character {

case dog = “🐶”, cat = “🐱”, rabbit = “🐰”, turtle = “🐢”

}

 

通常情况下，在性格测试的结尾，玩家会收到一些有关测试结果的文本。因为你已经定义了 enum 用来表示每种性格类型（在本例中为动物类型），那么你可以创建一个 definition 属性，它将在结果屏幕上显示为标签。

这里是动物类型的 definition 的示例：

 

enum AnimalType: Character {

case dog = “🐶”, cat = “🐱”, rabbit = “🐰”, turtle = “🐢”

 

var definition: String {

switch self {

case .dog:

return “You are incredibly outgoing. You surround

yourself with the people you love and enjoy

activities with your friends.”

case .cat:

return “Mischievous, yet mild-tempered, you enjoy

doing things on your own terms.”

case .rabbit:

return “You love everything that’s soft. You are

healthy and full of energy.”

case .turtle:

return “You are wise beyond your years, and you

focus on the details. Slow and steady wins the race.”

}

}

}

 

显示问题和答案

QuestionViewController 将在名为 questions 的属性中容纳 Question 对象的数组。在创建对象时，需要特别注意在 answers 属性中放置了多少个 Answer 对象。在为单个和多个答案响应构建堆栈视图时，你创建了四个按钮和四个开关，表示可能答案的静态数量。所以，如果 type 属性已设置为 single 或 multiple，使用该属性创建的所有 Question 都必须恰好有四个 Answer 对象。

范围响应可以只有两个答案，分别设置在滑块的两端。但如果定义四个可能的范围就更好，因为可以根据问题的四种结果分别打分。范围响应的答案集合需要依照某种顺序（比如，从最不符合到最符合），这样才能精确地将答案对应到结果。

在下面的示例中，数组中包含了每种响应类型的问题：单个答案、多个答案、范围响应。

 

var questions: [Question] = [

type: .single,

answers: [

Answer(text: "Fish", type: .cat),

Answer(text: "Carrots", type: .rabbit),

Answer(text: "Corn", type: .turtle)

]

),

 

Question(

text: "Which activities do you enjoy?",

type: .multiple,

answers: [

Answer(text: "Swimming", type: .turtle),

Answer(text: "Sleeping", type: .cat),

Answer(text: "Cuddling", type: .rabbit),

Answer(text: "Eating", type: .dog)

]

),

 

Question(

text: "How much do you enjoy car rides?",

type: .ranged,

answers: [

Answer(text: "I dislike them", type: .cat),

Answer(text: "I get a little nervous", type: .rabbit),

]

)

]

 

使用合适的控件显示问题

既然问题列表已经准备好了，那就需要跟踪 app 已经显示了哪些问题，计算何时显示完所有问题。方法之一是使用整数作为 questions 集合中的索引。此整数将从 0 开始（集合中第一个元素的索引），玩家回答完每个问题之后，该值会按 1 递增。

将一个名为 questionIndex 的属性添加到 QuestionViewController：

 

var questionIndex = 0

 

当玩家从一个问题移动到下一个问题时，你需要显示正确的堆栈视图并隐藏另外两个视图。但你需要先创建必要的插座变量和响应动作，然后才能编写代码来更改堆栈视图可见性。

 

 

打开 Main.storyboard 并选择 QuestionViewController。打开助理编辑器以将 QuestionViewController.swift 与 Storyboard 一起查看。按住 control 键，从单个答案堆栈视图拖移到 QuestionViewController 类的定义，然后松开鼠标或触控板，以显示弹出窗口。验证“连接”类型是否已设置为“插座变量”，然后在“名称”字段中输入“singleStackView”并点按“连接”。再重复两次这些步骤，为多个答案堆栈输入名称 multipleStackView，为范围堆栈输入 rangedStackView。

接下来，创建可重复使用的方法 updateUI()，可以在向玩家显示每个问题之前调用该方法。你应该在 viewDidLoad() 中调用此方法以针对第一个问题设置合适的界面。

 

override func viewDidLoad() {

super.viewDidLoad()

updateUI()

}

 

func updateUI() {

 

}

updateUI() 方法负责更新界面的几个关键部分，包括导航栏中的标题和堆栈视图的可见性。你可以使用 questionIndex 属性在导航项中为每个问题创建唯一的标题（例如“Question #4”）。利用堆栈视图，最简单的方法就是先隐藏全部三个堆栈视图，然后检查 Question 的 type 属性，以确定应显示哪个堆栈。

你可以结合使用 questionIndex 属性和 questions 集合来访问特定问题：

 

func updateUI() {

singleStackView.isHidden = true

multipleStackView.isHidden = true

rangedStackView.isHidden = true

 

navigationItem.title = “Question #\(questionIndex + 1)”

 

let currentQuestion = questions[questionIndex]

 

switch currentQuestion.type {

case .single:

singleStackView.isHidden = false

case .multiple:

multipleStackView.isHidden = false

case .ranged:

rangedStackView.isHidden = false

}

}

 

构建并运行 app。如果你的所有设置都正确，那么可见的堆栈视图应该与你在 questions 属性中定义的第一个问题相对应。尝试重新排列问题顺序以测试每个界面。

 

更新按钮和标签文本

问题屏幕上的界面可正常工作，但仍需要更新按钮标题和标签文本。要实现此目的，你需要为每个堆栈视图所关联的标签和按钮创建插座变量。

除了为堆栈视图创建的插座变量之外，此屏幕还需要 12 个用于控件和标签的插座变量。单个答案堆栈视图中有四个按钮插座变量，多个答案堆栈视图中有四个标签插座变量，而范围响应堆栈视图中有两个标签插座变量。在屏幕顶部附近还有显示问题文本的标签，底部附近有显示进度的视图。

在你创建大量插座变量时，必须使用简明又容易识别的变量名，还要让变量声明位于其对应的堆栈视图插座变量附近。下面的代码提供了很好的变量名和插座变量组织的示例：

 

@IBOutlet var questionLabel: UILabel!

 

@IBOutlet var singleStackView: UIStackView!

@IBOutlet var singleButton1: UIButton!

@IBOutlet var singleButton2: UIButton!

@IBOutlet var singleButton3: UIButton!

@IBOutlet var singleButton4: UIButton!

 

 

@IBOutlet var multipleStackView: UIStackView!

@IBOutlet var multiLabel1: UILabel!

@IBOutlet var multiLabel2: UILabel!

@IBOutlet var multiLabel3: UILabel!

@IBOutlet var multiLabel4: UILabel!

 

 

@IBOutlet var rangedStackView: UIStackView!

@IBOutlet var rangedLabel1: UILabel!

@IBOutlet var rangedLabel2: UILabel!

 

@IBOutlet var questionProgressView: UIProgressView!

继续创建上面的插座变量。因为屏幕中有很多控件相互重叠，所以你可能会发现，与其按住 control 键并从项拖移到画布上，不如按住 control 键并从文稿大纲中的项拖移到视图控制器定义，这样更简单。

插座变量准备好后，就可以更新 updateUI() 方法中的每个控件了。有两个插座变量 questionLabel 和 questionProgressView 需要随着每个新问题进行更新。只有在要显示标签和按钮插座变量的相关堆栈视图时，才需要更新这些插座变量。

对于问题标签，将其文本分配给当前问题字符串。对于进度视图，用 questionIndex 除以问题总数来计算出进度百分比。

 

func updateUI() {

singleStackView.isHidden = true

multipleStackView.isHidden = true

rangedStackView.isHidden = true

 

let currentQuestion = questions[questionIndex]

let currentAnswers = currentQuestion.answers

let totalProgress = Float(questionIndex) /

Float(questions.count)

 

navigationItem.title = “Question #\(questionIndex + 1)”

questionLabel.text = currentQuestion.text

questionProgressView.setProgress(totalProgress, animated:

true)

 

switch currentQuestion.type {

case .single:

singleStackView.isHidden = false

case .multiple:

multipleStackView.isHidden = false

case .ranged:

rangedStackView.isHidden = false

}

}

为了保持 switch 语句的简洁，可以对堆栈特定控件的更新进行重构（采用这些控件自己的方法)。

 

在单个答案堆栈视图中，每个按钮标题都对应一个答案。使用 setTitle(_:for:) 方法来更新标题。第一个按钮将使用第一个答案字符串，第二个按钮将使用第二个答案字符串，依此类推。

 

func updateSingleStack(using answers: [Answer]) {

singleStackView.isHidden = false

singleButton1.setTitle(answers[0].text, for: .normal)

singleButton2.setTitle(answers[1].text, for: .normal)

singleButton3.setTitle(answers[2].text, for: .normal)

singleButton4.setTitle(answers[3].text, for: .normal)

}

 

与之类似地，在多个答案堆栈视图中，每个标签的文本都对应一个答案。将第一个标签的 text 属性设置为第一个答案字符串，对另外三个标签重复此操作。

 

func updateMultipleStack(using answers: [Answer]) {

multipleStackView.isHidden = false

multiLabel1.text = answers[0].text

multiLabel2.text = answers[1].text

multiLabel3.text = answers[2].text

multiLabel4.text = answers[3].text

}

 

对于范围响应，堆栈视图的设置略有不同。由于只有两个要更新的标签，所以如果每个问题都有四个答案（虽然最少只需要两个答案），那么测试会运行得更好。

因为答案数不是一定的，所以直接在集合中使用索引的方法并不安全。例如，如果你使用了 answers[3] 来访问 answers 的第四个元素，但集合中仅包含两个 Answer 结构体实例元素，那么程序将崩溃。

 

无论你的范围响应问题有多少个答案，集合的 first 和 last 属性都允许你安全地访问对应于标签的两个 Answer 结构体。

 

func updateRangedStack(using answers: [Answer]) {

rangedStackView.isHidden = false

rangedLabel1.text = answers.first? .text

rangedLabel2.text = answers.last? .text

}

 

在定义这三个新方法的情况下，更新 updateUI() 中的 switch 语句 case 以调用这些方法。

 

switch currentQuestion.type {

case .single:

updateSingleStack(using: currentAnswers)

case .multiple:

updateMultipleStack(using: currentAnswers)

case .ranged:

updateRangedStack(using: currentAnswers)

}

 

构建并运行 app。标签和按钮都应该更新以反映第一个问题。干得漂亮！如果你看到控件未更新，请使用 QuestionViewController 的“Connections”检查器来验证是否已正确创建每个 IBOutlet。悬停在列表中的每项上方以检查其插座变量，如果需要，还可以修改任何不正确的插座变量。

使用响应动作接收答案

到目前为止，你已经生成了问题列表，app 正在显示第一个问题的正确用户界面。这是个很不错的开始。在这一部分中，你会记录玩家的答案并转至下一个问题。当玩家回答完集合中的每个问题之后，就会显示结果屏幕。

无论你使用哪组控件，都需要初始化一个空集合，以便储存玩家的答案。

 

var answersChosen: [Answer] = []

 

在单个答案堆栈视图中，你要决定每次轻点按钮对应哪个结果，将其附加到集合，然后继续下一个问题。全部四个按钮都会执行相同的工作，这样你就可以创建一个 IBAction，在轻点四个按钮中的任意一个时都会调用该相应动作。

 

开始时，按住 control 键并从单个答案堆栈视图中的第一个按钮拖移到 QuizViewController 类定义中的空间。这跟创建 IBOutlet 的方式相同，但这一次，你要将“连接”类型更改为“响应动作”。将该方法命名为“singleAnswerButtonPressed”并将“类型”从“Any”更改为“UIButton”，这样该方法的 sender 参数就会是 UIButton 类型。

 

 

 

将单个答案堆栈视图中的另外三个按钮连接到此新创建的 IBAction。

为什么需要更新类型？因为你需要将来自多个按钮的轻点动作都对应到这一个响应动作，所以需要指明哪个按钮触发了该方法。可以使用 if 语句和 == 来比较两个 UIButton 对象，也可以使用 switch 语句。如果该方法是使用 singleButton1 触发的，那么 app 就会知道玩家选择了第一个答案。

尝试在响应动作主体中填充代码，用于检查点击了哪个按钮，然后将所选答案附加到 answersChosen 数组。将答案添加到数组完成后，就需要继续到下一个问题。创建名为 nextQuestion() 的新方法并将其保留为空，后面将填充其内容。目前只需在 singleAnswerButtonPressed(_:) 的末尾添加 nextQuestion() 的调用。

@IBAction func singleAnswerButtonPressed(_ sender: UIButton) {

let currentAnswers = questions[questionIndex].answers

 

switch sender {

case singleButton1:

answersChosen.append(currentAnswers[0])

case singleButton2:

answersChosen.append(currentAnswers[1])

case singleButton3:

answersChosen.append(currentAnswers[2])

case singleButton4:

answersChosen.append(currentAnswers[3])

default:

break

}

 

nextQuestion()

}

对于多个答案的用户界面，你要根据玩家启用的开关来决定将哪些答案添加到集合中。按住 control 键并从“提交答案”按钮拖移到代码，然后使用“multipleAnswerButtonPressed”名称创建一个响应动作。继续，将“自变量”属性更改为“无”，因为你不需要该按钮来确定选择了哪些答案。

 

接下来，创建四个插座变量，每个 UISwitch 对应一个，这样你就可以检查哪些已启用，然后将那些答案添加到集合中。按住 control 键并从文稿大纲中的每个 UISwitch 拖移到代码，然后为每个开关指定名称。为了保持代码整洁有序，为多个答案堆栈视图相关联的标签变量附近的每个插座变量输入代码。

 

@IBOutlet var multiSwitch1: UISwitch!

@IBOutlet var multiSwitch2: UISwitch!

@IBOutlet var multiSwitch3: UISwitch!

@IBOutlet var multiSwitch4: UISwitch!

 

如果启用了第一个开关，你要添加第一个答案。与单个答案问题的方法不同，这一方法允许为每个问题附加多达 4 个答案。

 

@IBAction func multipleAnswerButtonPressed() {

let currentAnswers = questions[questionIndex].answers

 

if multiSwitch1.isOn {

answersChosen.append(currentAnswers[0])

}

if multiSwitch2.isOn {

answersChosen.append(currentAnswers[1])

}

if multiSwitch3.isOn {

answersChosen.append(currentAnswers[2])

}

if multiSwitch4.isOn {

answersChosen.append(currentAnswers[3])

}

 

nextQuestion()

}

对于范围响应问题，你要读取 UISlider 的当前位置并使用该值来确定将哪个答案添加到集合中。按住 control 键并从“提交答案”按钮拖移到代码，然后使用“rangedAnswerButtonPressed”​
名称创建一个响应动作。再次将“Arguments”属性更改为“None”。

 

接下来，为 UISlider 创建一个 IBOutlet。按住 control 键并从文稿大纲中的滑块拖移到代码，并指定名称。跟在之前步骤中的操作一样，将此插座变量的代码放置在与范围响应堆栈视图相关联的标签变量附近。

 

@IBOutlet var rangedSlider: UISlider!

 

花点时间考虑一下如何使用滑块的值来对应四个不同的答案。滑块的值范围从 0 到 1，所以 0 到 0.25 之间的值可对应第一个答案，而 0.75 到 1 之间的值可对应最后一个答案。

为了将滑块值转换为数组的索引，请使用 index = slider value * (number of answers - 1) 等式四舍五入为最接近的整数。这样会为 rangedAnswerButtonPressed 实现以下方法：

 

@IBAction func rangedAnswerButtonPressed() {

let currentAnswers = questions[questionIndex].answers

let index = Int(round(rangedSlider.value *

Float(currentAnswers.count - 1)))

 

answersChosen.append(currentAnswers[index])

 

nextQuestion()

}

将数据传递到结果视图

在显示结果屏幕时，需要向其传递用户的响应。打开 ResultsViewController.swift 并添加用于存储用户响应的属性。

 

var responses: [Answer]

 

这将导致 Xcode 报告编译错误，该错误为“Class ‘ResultsViewController’ has no initializers”。这是因为 responses 没有初始值，也没有在构造器中为其赋值。要解决此问题，你要创建 ResultsViewController 的自定构造器，采用 responses 作为自变量，从而满足编译器要求。添加以下构造器：

 

init?(coder: NSCoder, responses: [Answer]) {

self.responses = responses

super.init(coder: coder)

}

 

这个新的构造器采用两个参数：coder 和 responses。coder 参数将由 UIKit 提供并用于根据 Storyboard 中定义的信息初始化视图控制器。responses 参数将在调用此构造器时由你提供，并赋给你刚才添加的 self.responses。最后，super 构造器会在通过 coder 传递时被调用。

 

不幸的是，所有这些努力导致了新的编译错误，该错误为“required initializer init(coder:) must be provided by subclass of UIViewController.”。提供你自己的构造器时，必须实现超类定义的所有 required 构造器。

有关更多信息，请点按错误旁边的红色符号。

Xcode 提供了此问题的“修复方法”。点按“Fix”按钮以插入建议的代码修复。

 

required init?(coder: NSCoder) {

fatalError(”init(coder:) has not been implemented”)

}

 

在本例中，你不必关注于代码提供的 required 构造器，因为你将使用自己的构造器。现在，required 构造器的这部分代码在调用时会使应用程序崩溃，你不可以调用该构造器。

对已回答问题的响应

既然你已经处理了每个问题的用户输入，并可以将响应传递到 ResultsViewController，现在该实现 QuestionViewController 中的 nextQuestion() 方法了。为此，你将 questionIndex 的值递增 1，然后确定是否还剩余任何问题。如果有，则调用 updateUI() 来更新标题并显示适当的堆栈视图。该方法将使用新的 questionIndex 值来显示下一个问题。如果没有剩余问题，现在该使用 Storyboard 中创建的过渡来展示结果了。

 

func nextQuestion() {

questionIndex += 1

 

if questionIndex < questions.count {

updateUI()

} else {

performSegue(withIdentifier: “Results”, sender: nil)

}

}

第 5 部分

计算并显示结果

你有一个针对多种问题类型的有效用户界面，还可以记录玩家的答案。最后的步骤包括计算结果、显示结果以及关闭结果屏幕（以便新的玩家可以开始测试）。

计算答案频率

既然 ResultsViewController 有了玩家的响应，那么接下来是时候考虑一下如何计算性格了。在所选答案中，哪种性格类型最常见？在此示例中，如果玩家给出两个高度对应狗的答案，而仅给出一个对应其他动物的答案，那么最佳结果就是狗。如果大部分答案都指向两种或多种动物，那么哪种动物都是有效的结果。

如何统计结果？你需要循环遍历 responses 属性中的每个 Answer 结构体，并计算集合中哪种 type 是最常见的。好的解决方案可能是字典数据模型，其中，键是响应类型，而值是玩家将其选为答案的次数，这实际上就是一个直方图。例如，如果玩家给出两个与狗高度对应的答案，而只给出一个与其他每种动物对应的答案，那么字典会如下所示：

 

{

cat : 1,

dog : 2,

rabbit : 1,

turtle : 1

}

 

首先，在 ResultsViewController.swift 中，声明一个你可以在 viewDidLoad() 中调用的名为 calculatePesonalityResult 的方法。在该方法中，你将计算每个响应的频率，如以上代码中所示。对于“你是哪种动物？”测试，可以使用以下代码：

 

override func viewDidLoad() {

super.viewDidLoad()

calculatePersonalityResult()

}

 

func calculatePersonalityResult() {

let frequencyOfAnswers = responses.reduce(into: [:]) {

(counts, answer) in

counts[answer.type, default: 0] += 1

}

}

 

计算结果时，不需要整个 Answer 结构体。你只关心每个 Answer 的 type 属性。因此，可以将 responses 缩减为 [AnimalType: Int] 类型的新字典，其中 Int 是玩家选择对应 AnimalType 的频率。

下面是发生的情况的详细分解。reduce(into:) 是 Array 上的一种方法，用于对每个项迭代，并使用你提供的代码将其组合为单个值。正如 for...in 循环会为数组中的每个项执行其主体一次，reduce(into:) 会针对每个项执行以下花括号中的代码一次。但是，花括号中的代码不同于循环主体。它是采用两个参数的闭包，显示为 (counts, answer) in。第一个参数是你要缩减为的项。第二个参数是集合中用于当前迭代的项。（可以在 Swift 编程语言指南中找到有关闭包的更多信息。）

在本例中，你要将 responses 缩减为初始为空的字典（为 into: 传递的自变量），然后将字典中对应于每个响应的键的值递增。注意 counts 的下标中的 default: 参数。通过这种方式可防止字典中缺少键。如果键不存在，会创建该键并将其设置为针对 default 指定的值。通过提供默认值，就可以避免原本需要额外逻辑来处理键不存在时，字典通常会返回 nil 值的情况。

下面是不使用 default 下标语法时的等效代码：

let frequencyOfAnswers = responses.reduce(into:

[AnimalType: Int]()) { (counts, answer) in

if let existingCount = counts[answer.type] {

counts[answer.type] = existingCount + 1

} else {

counts[answer.type] = 1

}

}

 

如果不为不存在的键指定默认值，就必须尝试在它递增前解包下标结果，如果尚不存在，就插入新值。还请注意，现在你需要为 into: 参数的初始值指定类型信息。将 0 指定为 counts 的默认值时，Swift 能够推断出 counts 字典的值应该为 Int 类型。（在两种情况下，Swift 都可以推断出键是 AnimalType，因为这是你传递的类型。）

有关更多详细信息，按住 option 键并点按 reduce(into:) 以打开快速帮助。你将找到刚好是你刚才所执行操作的示例用法，这是该任务的理想方法。

确定出现频率最高的答案

既然已经有了知道每个响应频率的字典，那么接下来就可以确定哪个值最大了。你可以对字典使用 Swift 编程中的 sorted(by:) 方法，将每个键值对放入数组中，并按 value 属性的降序排列。

 

let frequentAnswersSorted = frequencyOfAnswers.sorted(by:

{ (pair1, pair2) in

return pair1.value > pair2.value

})

 

let mostCommonAnswer = frequentAnswersSorted.first! .key

 

此代码具体怎样工作呢？假定你的字典如下所示：

 

{

cat : 1,

dog : 2,

rabbit : 1,

turtle : 1

}

 

你传递到 sorted(by:) 中的参数是可接受任意两个键值对的闭包。在动物测试中，pair1 可能对应 cat : 1，pair2 可能对应 dog : 2。在闭包的正文中，需要返回布尔值以指示哪对更大。如果 return 1 > 2，那么布尔值为 false，所以该方法就会知道 pair2 比 pair1 大。

在该方法结束后，数组 frequentAnswersSorted 可能看起来与下面的代码类似。对于值相同的键值对，没办法进行排名，所以兔子、乌龟和猫的顺序可能会不同。但是没关系，因为你只关心数组中的第一个元素。

 

[(dog, 2), (rabbit, 1), (turtle, 1), (cat, 1)]

 

可以使用四个技巧简化闭包。首先，可以使用 $0 和 $1 作为隐式参数名称，而不定义你自己的参数，这样就可以删除 (pair1, pair2) in。还可以按数字而不是按名称（将 .value 的引用替换为 1）引用这些对的元素。接下来，可以为闭包消除只包含一个表达式的 return 语句。闭包会自动返回该值。最后，可以使用尾随闭包语法删除对 sorted(by:) 的调用周围的括号，并消除自变量标签。上述对 reduce(into:) 的调用也使用了尾随闭包语法。该方法有两个参数。你需要在括号中使用传统的函数调用语法来传递第一个自变量。然后，你在括号之外提供了最后一个自变量，即尾随闭包。

通过使用这些技巧并检索结果的第一个元素的键，可以将代码简化为一行：

 

let mostCommonAnswer = frequencyOfAnswers.sorted { $0.1 >

$1.1 }.first! .key

 

查看结果

重新启动测试

在大多数性格测试中，玩家都只会将所有问题回答一遍。在显示结果后，玩家不应该能够返回并更改先前所答问题以尝试得到不同的结果。但可惜结果屏幕上的“Back”按钮暗示玩家可以这样做。要隐藏导航栏中的“Back”按钮，请将以下代码行添加到 ResultsViewController 的 viewDidLoad() 的底部：

 

navigationItem.hidesBackButton = true

 

玩家应该可以关闭结果并重新开始，而不是更改先前的响应。轻点“Done”按钮会返回 IntroductionViewController，这会清晰地表明测试已结束。但那时候，“Done”按钮不连接任何种类的响应动作。

你需要在第一个视图控制器中创建回退方法。将以下内容添加到 IntroductionViewController 定义中：

 

@IBAction func unwindToQuizIntroduction(segue:

UIStoryboardSegue) {

 

}

 

因为在关闭 ResultsViewController 时，app 不需要保留或传回任何信息，所以这个方法的内容可以先为空白。

在 Storyboard 中，按住 control 键并从“Done”按钮拖移到视图控制器顶部的“退出”按钮。选中在弹出窗口中显示的 unwindToQuizIntroductionWithSegue 选项。

 

现在，当玩家轻点“Done”按钮时，回退过渡将关闭在 IntroductionViewController 显示之后创建的视图控制器。包括 QuestionViewController 和 ResultsViewController。
