既然你已经学习了 Interface Builder 的基本知识，那么应该知道视图控制器是 app 内部结构的基础。每个 app 都有至少一个视图控制器，大多数 app 有数个视图控制器。

本节课将解释有关视图控制器生命周期的更多信息，让你可以了解这一重要类的潜力。

  • 如何在视图层次结构中添加和删除视图

在设置完成视图和视图层次结构之后，还需要继续编写 app 背后的逻辑。你的视图控制器类负责显示用户的数据，还负责处理用户交互。在“灯光”项目中，你使用 UIViewController 子类通过响应动作来管理按钮轻点事件，还相应地更新了视图的背景颜色。视图控制器还可以控制视图的创建，可以根据视图在其生命周期中的进度状态来处理事件，可以在不再需要视图之时释放视图。

 

视图控制器生命周期

在 iOS 中，视图控制器可以处于几种不同的状态：

  • 视图未载入
  • 视图正在显示
  • 视图已显示
  • 视图正在消失
  • 视图已消失

当视图从一种状态转换为另一种状态时，iOS 会调用 SDK 定义的方法，此过程可以在代码中得以实现。在下图中，你可以看到在发生状态转换时调用的每个方法的名称。

你可能注意到了方法名称中的规律。在载入视图之后，方法成对出现：“will”和“did”。（请比较变量的 willSet 和 didSet 属性观察器的名称。）这一 Apple 标准设计式样可用于在发生指定事件之前和之后编写代码。但必须要注意，可能会发生有“will”回调但没有对应“did”回调的情况。例如，即使从未发生过 viewDidDisappear(_:)，也可以调用 viewWillDisappear(_:)。

每个 app 都是不同的，你会学习如何处理每种视图控制器生命周期方法，以及与其关联的特定任务类型。

为了获得处理视图控制器生命周期的实际操作经验，你要构建一个在控制台中打印消息的 app，以在调用每个生命周期方法时提供说明。随着你进一步了解不同的生命周期方法，你将可以将方法添加到项目中。

在 Xcode 中，使用 iOS“App”模板创建新的项目。将项目命名为“LifeCycle”。

 

视图已载入

在完成视图控制器实例化之后（无论是从 Storyboard 还是以编程方式），视图控制器都会将视图载入内存。此过程会创建将由控制器管理的视图。

在视图控制器完成载入特定视图之后，会调用其对应的 viewDidLoad() 函数，让控制器有机会针对已载入并就绪的视图执行工作。例如，在“灯光”项目中，会调用 updateUI() 来同步 lightOn 状态下的视图背景颜色。要在 viewDidLoad() 中执行的其他设置任务类型包括：额外的视图初始化、网络请求、数据库访问。

如果要添加视图控制器生命周期方法的自定实现，需要知道与该视图控制器场景关联的类。在 LifeCycle 中，打开 Main.storyboard 以显示其视图控制器。选中该视图控制器，并使用“Identity”检查器来显示其子类。

 

 

在“Class”字段中，你会看到灰色圆圈中有一个白色箭头，这是访问该类的快捷方式。点按该箭头可以打开关联的 .swift 文件 ViewController.swift。

 

将以下内容添加到 viewDidLoad() 函数的末尾：

 

print(”ViewController - View Did Load”)

 

只要调用 viewDidLoad()，此行就会在控制台中打印字符串“ViewController - View Did Load”。

 

构建并运行 app。载入视图后，应该会在控制台面板中看到你的消息。

 

视图事件管理

有些工作单元可能只会执行一次，如更新标签的字体、文本或颜色。viewDidLoad() 最适合用来执行这些任务。

对于那些需要执行多次的工作，app 可以依赖视图事件通知。如果视图可见性发生了改变，那么视图控制器会自动调用其生命周期方法，这样你就可以对视图状态的改变做出响应。

这些方法包括：

  • viewWillAppear(_:)
  • viewDidAppear(_:)
  • viewWillDisappear(_:)
  • viewDidDisappear(_:)

请阅读文档中的相关说明。你会发现这些方法中每一种方法都需要在实现过程中的某个时间点调用超类的版本。有一种方式可以了解为什么需要发生此操作，请思考一下这种情况：当你编写自定视图控制器子类并想要使用生命周期方法（包括 viewDidLoad()）时，如果不使用 override 关键字，就会发生错误。这是因为超类中已经包含了这些方法的定义。使用 override 关键字覆盖超类（在本例中为 UIViewController）中的实现，并且允许执行此方法的实现，而不允许执行超类的实现。

但是，因为你不知道 UIViewController 的生命周期方法的实现，所以如果不让超类的代码运行，可能会发生意想不到的问题。UIViewController 所执行的工作至关重要，app 需要它才能正常发挥功能。为了修正此问题，可以使用 super 关键字明确地调用该方法的超类版本。现在你的代码和 UIViewController 的代码都会运行。

对超类实现的调用通常都会是覆盖方法的第一行。

示例：

 

override func viewWillAppear(_ animated: Bool) {

super.viewWillAppear(animated)

// 在这里添加你的代码

}

 

 

视图将显示和视图已显示

在 viewDidLoad() 之后，视图控制器生命周期中的下一个方法是 viewWillAppear(_:)。此函数会在屏幕上即将显示视图时被调用。此处非常适合添加需要在每次向用户显示视图之前执行的工作。例如，如果视图显示与用户位置相关的信息，那么你可能要在 viewWillAppear(_:) 中请求位置。这样，视图就可以更新以利用新位置。其他任务包括：开始网络请求，刷新或更新视图（如状态栏、导航栏或表格视图），以及调整以适应新的屏幕方向。

正如你所预期的那样，视图显示在屏幕上之后将调用 viewDidAppear(_:)。如果需要在每次视图显示时都执行某项工作，且工作用时可能不只几秒钟，那么应将该工作置于 viewDidAppear(_:) 中。这样，视图会快速显示出来，而函数也将继续执行。

应使用 viewDidAppear(_:) 方法来启动动画或其他长时间运行的代码，如获取数据。

如果要继续尝试使用生命周期，请覆盖 viewWillAppear(_:) 函数。

 

override func viewWillAppear(_ animated: Bool) {

 

}

 

因为你要编写属于自己的 viewWillAppear(_:) 自定实现，所以要记得调用 viewWillAppear(_:) 的超类版本。因为调用 viewWillAppear(_:) 需要有 animated 属性，所以可以一并传送为子类指定的参数。

 

override func viewWillAppear(_ animated: Bool) {

super.viewWillAppear(animated)

}

视图将消失和视图已消失

你可能已经猜到了，在视图从屏幕中消失之前会调用 viewWillDisappear(_:)。当用户轻点后退按钮、切换标签、显示或关闭模态屏幕时，就会离开当前屏幕，此时将执行此方法。你可以使用 viewWillDisappear(_:) 方法来存储编辑、隐藏键盘或取消网络请求。

生命周期中的最后一个方法是 viewDidDisappear(_:)，当视图从屏幕中消失后会调用此方法（通常发生在用户导航到新视图之后）。此方法一旦执行，就可以确定视图已经消失。所以说，此方法让你有机会停止与视图有关的服务，如播放视频或删除通知监视程序。

现在花一点时间返回“LifeCycle”项目。在 ViewController 类中，添加并覆盖消失视图事件的“will”和“did”函数。添加打印说明以便看到生命周期事件。

要让视图消失，需要再添加一个视图控制器。否则就无法离开当前视图。但首先，要添加一个用于处理导航的标签栏控制器。

 

 

为了查看 iOS 如何在视图控制器之间转换，也要使用 SecondViewController 的生命周期方法。返回 SecondViewController 类文件，添加与 ViewController 中相同的五个生命周期方法。别忘了调用 super 实现。为每个事件添加打印说明，例如：“SecondViewController - 视图已载入”。

构建并运行 app。无需任何点按，就应该能看到 ViewController 实例已载入，且两个显示方法都已调用。

可是为什么看不到 SecondViewController 有任何打印内容呢？Storyboard 初始化了 SecondViewController，然后会发生什么？为什么没载入视图？视图控制器只有在需要将视图显示给用户时才会载入视图。这是非常好的事情。通过延迟载入，app 可以节省内存，毕竟在移动环境中，内存资源有限。

如果要载入 SecondViewController 实例，请返回“Simulator”，然后点按第二个视图控制器的标签。此时你应该会看到按此顺序显示的下列打印说明：

ViewController - 视图已载入

ViewController - 视图将显示

ViewController - 视图已显示

SecondViewController - 视图已载入

SecondViewController - 视图将显示

ViewController - 视图将消失

ViewController - 视图已消失

SecondViewController - 视图已显示

切换回第一个标签，打印说明的顺序如下：

ViewController - 视图将显示

SecondViewController - 视图将消失

SecondViewController - 视图已消失

ViewController - 视图已显示

调用这些函数的顺序有助于表明在视图层次结构中添加和删除视图控制器视图的方式：

  • 如果要将视图控制器的视图添加到层次结构中，UIKit 会首先确保已载入该视图。如果视图尚未载入，它会载入该视图并触发 viewDidLoad()。
  • 在将视图控制器的视图添加到层次结构之前，UIKit 会先触发 viewWillAppear(_:)。
  • 不再显示的视图控制器视图将被删除，并将调用这些视图控制器的 viewWillDisappear(_:) 和 viewDidDisappear(_:) 方法。
  • 最后，UIKit 会显示新视图并触发 viewDidAppear(_:)。

 

如你所见，视图控制器生命周期方法有很多用法。每个方法都像是一种“通知”，告诉你的代码，视图事件已发生。使用此指南，开发者将学会如何最好地利用每个方法来处理手头的特定任务。

练习：事件的顺序

目标

本练习的目标是帮助你进一步了解视图的生命周期。你将创建一个 app，用于根据视图控制器生命周期中的事件来添加标签文本。

使用 iOS“App”模板创建一个名为“OrderOfEvents”的新项目。

第 1 步

创建新的视图控制器子类

  • 再从“Object”库中拖出两个视图控制器，将其放在 Storyboard 中现有视图控制器的旁边。第一个视图控制器只是 app 的起点，它包含一个按钮，用于过渡到下一屏幕。第二个视图控制器包含一个标签，用于显示不同视图控制器生命周期事件的顺序。第三个视图控制器将提供离开第二个视图控制器的方式。
  • 通过在 Xcode 菜单栏中选择“File”>“New”>“File”（或按 command-N），然后选择“Cocoa Touch Class”，创建两个新的 .swift 文件，分别用于两个新的视图控制器。将文件命名为“MiddleViewController”和“LastViewController”。检查两个文件是否都是 UIViewController 的子类。
  • 在 Storyboard 中选择中间和最后一个视图控制器并将其链接到刚创建的文件（每次操作一个文件），使用“Identity”检查器将其类设置为 MiddleViewController 和 LastViewController。
  • 请注意，Storyboard 中的原始视图控制器（即项目模板提供的控制器）在“Identity”检查器中已经有对应的 ViewController.swift 文件，并且已经正确指定其类。

第 2 步

设置 Storyboard

  • 将第一个视图控制器嵌入导航控制器中。
  • 在此视图控制器上，添加一个文本为“Show me the life cycle”的按钮。创建一个从该按钮到中间视图控制器的 Show 过渡。
  • 在中间的视图控制器的底部添加一个按钮。创建一个从此按钮到最后一个视图控制器的 Show 过渡。因为两个视图控制器之间没有信息传递，所以不需要担心设置过渡的标识符。
  • 在中间的视图控制器的顶部添加一个标签。将其文本替换为“Nothing has happened yet”。因为此标签以后会需要更多行，所以请使用“Attributes”检查器将“Lines”设置为 0。此属性将使标签可以按需扩展。
  • 在最后一个视图控制器中添加一个标签，并将其文本替换为“Go back and see if anything happened”。

第 3 步

根据生命周期事件更新标签

  • 在 Storyboard 中，创建一个从中间视图控制器中的标签到 MiddleViewController.swift 的插座变量。
  • 在插座变量正下方为标签添加一个类型为 Int 的变量属性。将其命名为“eventNumber”，并设置为等于 1。在每个生命周期事件的末尾，代码将为此属性加 1，以此方式为添加到标签的事件指定编号。
  • 在 MiddleViewController.swift 中，添加一个方法，用于针对给定事件更新标签。使用条件绑定对标签中的现有文本进行解包。然后为标签文本设置已存在的文本，外加一条有关刚发生的生命周期事件的说明，然后更新 eventNumber。此说明应使用 eventNumber 来跟踪事件的顺序。代码可能类似于：

func addEvent(from: String) {

if let existingText = label.text {

label.text = "\(existingText)\nEvent number \(eventNumber) was \(from)"

eventNumber += 1

}

}

 

  • 发生了什么？以上代码对标签中的文本进行解包。你很有必要这样做，因为值是可选的。接下来，它会在标签文本中添加一个换行符 (\n)（另起一行）、有关刚发生的事件的描述，以及事件编号。代码会让 eventNumber 递增 1，所以下次访问 eventNumber 时，它就会按顺序描述下一个事件编号。
  • 在 MiddleViewController.swift 中，实现你在本节课中学到的全部五个生命周期方法：viewDidLoad()、viewWillAppear(_:)、viewDidAppear(_:)、viewWillDisappear(_:)、viewDidDisappear(_:)。

 

override func viewDidLoad() {

super.viewDidLoad()

addEvent(from: "viewDidLoad")

}

 

  • 在填充完全部五个生命周期方法的主体之后，运行 app 并在屏幕之间进行导航。如果导航到最后一个视图控制器，并返回中间的视图控制器时，则会发生什么情况？所有生命周期事件都会重新发生吗？如果不是，原因何在？如果一直导航回到初始视图控制器，然后再导航到中间的视图控制器，情况会怎样呢？所有生命周期事件都会重新发生吗？为什么此操作的结果与从最后一个导航到中间的视图控制器不同？
  • 非常好！你已经构建了一个 app，可以在发生视图控制器生命周期事件时显示其顺序。请务必将其存储到项目文件夹中。
